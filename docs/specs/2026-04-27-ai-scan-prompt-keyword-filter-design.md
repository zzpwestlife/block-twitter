# AI 扫描提示词优化：从备份关键词中筛选/分组高置信信号，降低误报并补漏

日期：2026-04-27  
范围：`src/content/content.js` → `AIDetector._getSystemPrompt()` / `AIDetector.DEFAULT_SYSTEM_PROMPT`  
参考数据：`block-twitter-backup-2026-04-27 (1).json`

## 背景 / 问题

当前 AI 扫描（`AIScanButton` → `AIDetector`）的误报率偏高，同时存在漏报。  
现有实现会将用户维护的 `keywords`（最多 80 条）直接追加到 system prompt：

- 优点：能给模型更多“垃圾账号线索”
- 风险：`keywords` 中可能混入正常句子/鸡汤/泛化短语，直接当作“垃圾示例”会抬高误报。

对备份 `keywords` 的粗分（仅供判断方向）：
- 总计 94 条
- 高置信类别占比较高：线下/同城/附近、主人/调教、明显色情、emoji-only、网盘链接
- 同时也存在低置信/泛化短语（如“共同兴趣，自我提升”等），不宜作为“垃圾示例”硬塞给模型

## 目标 / 成功标准

1. **降低误报**：正常内容不应因为“关键词示例污染”被判 spam。
2. **减少漏报**：在“未命中关键词高亮”的前提下，AI 仍能根据模式相似性识别垃圾账号。
3. **可控 Token**：避免把全部关键词原样塞入导致提示词膨胀。
4. 不改变现有 UI 流程与输出格式（仍然只输出 `spam` / `ok`）。

## 非目标

- 不新增复杂的模型训练/在线学习机制。
- 不改变 `KeywordMatcher` 的工作方式（本次仅优化 AI 扫描提示词构建）。

## 方案（采纳）

### 方案 A：关键词自动筛选 + 分组注入（采纳，推荐）

在构建 system prompt 时，不再使用 `keywords.slice(0, 80)` 的“原样拼接”，而是：

1. **对 keywords 做清洗与去重**
   - trim、去空、去重
2. **将 keywords 进行“高置信类别”筛选/分组**
   - `url`：包含 `http://` / `https://`（如 `https://pan.quark.cn`）
   - `sex`：明显色情/性服务相关词（如“固炮/招嫖/胸大/骚货/做爱”等）
   - `master`：主人/宠物/调教类（如“找主人/小狗/调教/等主”等）
   - `offline`：同城/线下/附近/地标（如“同城/线下/附近/万达广场/男大/体育生”等）
   - `emoji_only`：基本为 emoji/空白/换行组成、缺少实质文字（作为引流暗号信号）
3. **每个类别仅取 Top N（例如 10~15 条）注入**
4. **以“软规则/强信号”的方式写入提示词**
   - 明确说明：这是“高置信垃圾模式示例”，但仍需结合账号/内容整体判断
   - 对容易误伤的类别（如 offline）在提示词里附加约束：需要与引流/性暗示/求约等语境共同出现才判 spam

### 提示词文本调整（同属方案 A）

对 `AIDetector.DEFAULT_SYSTEM_PROMPT` 做结构化升级：

- 增加“保守原则”：证据不足时输出 `ok`（降低误报）
- 将 spam 规则拆为：
  - **强 spam**：色情服务、招嫖约炮、明显引流/诈骗、emoji-only 暗号、链接引流
  - **弱 spam（需组合证据）**：同城/线下/附近等词必须与求约/性暗示/引流共同出现
- 增加明确反例提示：正常观点/新闻/日常生活、正常 emoji 表达、正常地点讨论不应判 spam

## 实现要点

1. 在 `AIDetector._getSystemPrompt()` 中新增 `extractHighConfidenceKeywordGroups(keywords)`（或内联实现）：
   - 输入：`string[]`
   - 输出：`{ url: string[], sex: string[], master: string[], offline: string[], emoji_only: string[] }`
2. 拼接 prompt 时按类别写入，例如：

```text
【高置信垃圾模式示例（仅作参考，需结合语境）】
- 色情/性服务：…（最多 N 条）
- 主人/调教：…（最多 N 条）
- 同城/线下求约：…（最多 N 条，注意需结合求约/性暗示/引流）
- 外链引流：…（最多 N 条）
- 纯 emoji 暗号：…（最多 N 条）
```

3. 保持原有“最多 80”之类的硬限制思路，但改为“分类限额 + 全局限额”双限制，避免 token 失控。

## 测试用例（手动）

1. 使用同一批时间线内容进行 AI 扫描，观察：
   - 原先误报的账号是否减少（尤其是正常鸡汤/普通表达）
   - 原先漏报的典型模式账号是否更容易被识别
2. 对于仅出现“同城/附近”等词但无引流/性暗示的内容，期望判 `ok`。
3. 对于 emoji-only / 链接引流 / 主人调教 / 明显色情内容，期望判 `spam`。
